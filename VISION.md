# AIMeeBo Vision

## Core: AIMeeBo — AIを会議の「参加者」にする

### 現在の状態

会議の前後で動く受動的なツール（議事録生成・要約・タスク抽出）。

### 最終目標

**AIが会議のアクティブな参加者として、人間と対等に意思決定に関与する。**

会議中にリアルタイムで聞き、見て、考え、発言する。
質問されたら即座にデータを引き出し、根拠付きで回答する。
見落としを指摘し、選択肢を整理し、リスクを提示する。
ただし最終判断は常に人間に委ねる。

### ステージ

| Stage | レベル | 状態 |
|-------|--------|------|
| 0 | 会議の前後で動く（議事録・要約） | 完了 |
| 1 | 聞かれたら答える（リアルタイム音声理解 + ナレッジ検索） | 実装済み |
| 2 | 意思決定に参加する（自発的発言 + リスク分析 + 選択肢整理） | System Prompt 実装済み、検証中 |
| 3 | チームの一員として学習する（過去の会議を記憶、参加者の傾向を理解、組織のナレッジを蓄積） | 未着手 |

## Satellite: AIエコシステム — プロジェクト間をつなぐ知の循環

AIMeeBo は単体で完結しない。4つのプロジェクトがそれぞれの役割を持ち、つながることで価値が倍増する。

### 全体像

```
                    ┌──────────────────┐
                    │   ContextGit     │
                    │  (統合管理層)     │
                    │                  │
                    │  全プロジェクトの  │
                    │  コンテキストを   │
                    │  Git的に管理     │
                    └────────┬─────────┘
                             │
              コンテキスト共有│バージョン管理
                             │
    ┌────────────────────────┼────────────────────────┐
    │                        │                        │
    ▼                        ▼                        ▼
┌────────────┐      ┌───────────────┐      ┌──────────────┐
│   PBL      │      │   AIMeeBo     │      │  freelance   │
│ (知の構造化)│      │  (会議AI)     │      │  (営業効率化) │
│            │      │               │      │              │
│ 論文・研究  │─────→│ ナレッジベース │←─────│ 案件・進捗    │
│ マインドマップ│      │ 意思決定支援  │      │ クライアント  │
│ グラフDB   │      │ リアルタイム   │      │ データ       │
└────────────┘      └───────────────┘      └──────────────┘
     │                      │                      │
     └──────────────────────┼──────────────────────┘
                            │
                    ┌───────▼────────┐
                    │  共有される価値  │
                    │                │
                    │ ・統一されたコンテキスト │

```

### 各プロジェクトの役割

| プロジェクト | 一言での役割 | AIMeeBo との接続 |
|---|---|---|
| **PBL** | 知識を構造化する | 論文・研究データ → AIMeeBo のナレッジベース。会議中に「この疾患のエビデンスは？」と聞くと論文DBから回答。マインドマップで議論の構造を可視化。 |
| **ContextGit** | コンテキストを管理する | 会議のコンテキスト（議題・前回の決定・未解決事項）をGit的に管理。次回の会議で「前回どこまで話した？」が自動で復元される。複数AIの出力を統合管理。 |
| **freelance-toolkit** | 仕事を効率化する | 案件進捗・クライアントデータ → AIMeeBo のナレッジベース。プロジェクト打合せで「この案件の状況は？」に即答。 |

### つながりの具体例

**シナリオ: 研究プロジェクトの定例会議**

```
1. 会議開始
   → ContextGit が前回の会議コンテキストを復元
   → AIMeeBo が「前回の宿題は3つあります」と報告

2. 「心血管疾患の最新データある？」
   → AIMeeBo が PBL の論文DBを検索
   → 「2025年のRSAPの論文によると…」と根拠付きで回答
   → マインドマップで関連論文の構造を表示

3. 「このプロジェクトの予算消化率は？」
   → AIMeeBo が freelance-toolkit のトラッカーを参照
   → 「現在68%、予定通りです」と回答

4. 「来月の方針を決めよう」
   → AIMeeBo (Active Mode) が選択肢を整理
   → 「3つの選択肢があります。A案はリスクが低いがROIも低い…」
   → ContextGit に決定事項をコミット

5. 会議終了
   → 議事録・タスク・次回議題を自動生成
   → ContextGit に会議コンテキストをコミット
   → 次回の会議で全て自動復元
```

### 接続の技術的アプローチ

AIMeeBo 側は `KnowledgeSource` interface を実装済み。
各プロジェクト側にはAPIエンドポイントを追加するだけで接続可能。

```typescript
// AIMeeBo 側（実装済み）
interface KnowledgeSource {
  search(query: string, category?: string): Promise<KnowledgeResult[]>;
}

// 現在: MockKnowledgeBase
// 将来: PBLGraphDB, FreelanceTracker, ContextGitAPI
```

各プロジェクト側の改修は「接続したくなった時」に行う。
今の設計で後から接続できるようになっている。

## 根底にある思想

**「AIは道具ではなく、チームメンバーである」**

- 道具は使われるのを待つ。チームメンバーは自ら考えて動く。
- 道具は使い捨てられる。チームメンバーは文脈を共有し、学習する。
- 道具は単機能。チームメンバーは状況に応じて役割を変える。

AIMeeBo は、AIが人間のチームに溶け込むための最初の一歩。
会議という、人間が最も多くの時間を費やし、最も非効率な活動から始める。
